Running tests on Git pushes

Namespace includes a continuous integration and deployment system which knows how to build, test and deploy your Namespace-built services. But you can also your own CI to run tests.

Using Namespace's Continuous Integration​

We've built a Namespace optimized Continuous Integration that understands your service, server and test layout. It uses the service graph to present you with relevant information, and narrow detailed logs, which help you understand and debug problems you may hit.

When you setup Namespace's CI, you'll find:

  • Build and test workflows being executed automatically on changes to the main branch, or on pull requests: these workflows run in Namespace's own clusters, with high-parellism.
  • Updated Github status on git commits associated with the changes.
  • Detailed build and test logs available on the web, structured per run and per server.

To use our Continuous Integration:

  • Push your Namespace-built services live to a Github repository, either within your personal account, or in the account of an organization.
  • Grant our application access to your repository in https://github.com/apps/namespace-continuous-integration/installations/new
  • Contact us on Slack to enable the CI pipeline.
Github-only for now

We expect to support other providers in the future, but for the time being are limited to Github.

Paid feature later on

A heads up that our CI will be free during Early Access, but we expect it to become a paid feature later on.

Using your own CI​

You can rely on Github Actions, CircleCI, or your CI system of choice that includes Docker support, to run Namespace tests. If you do,

  • Namespace sets up a local development environment within your CI invocation, similar to what you'd use for local development.
  • Tests will be run within this production-like environment, just as they would otherwise.

To do so, create a workflow using your preferred CI system, and after cloning your repository, make sure you have the ns binary available.

After you do, add the following workflow steps:

  • ns prepare local
  • ns test --include_servers
  • ns deploy --naming_no_tls --wait
